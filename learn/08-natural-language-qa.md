**Goal:** understand how the text-to-SQL app works, run it against your real data,
and ask it questions in plain English.

## Concept

### The problem: SQL is a barrier

Your Gold marts now hold real, clean, queryable data. But to get an answer from
them you need to write SQL — which means knowing the table names, column names,
data types, and the right joins. That's a barrier for anyone who isn't a data
engineer.

**Text-to-SQL** removes that barrier: the user asks a question in plain English,
a language model translates it to SQL, the SQL runs against the real data, and
the model narrates the result.

### The four-step pipeline

```
1. build_schema_context()   → compact description of all Gold mart tables + columns
                               (read from dbt's catalog.json)

2. generate_sql()           → send (question + schema) to Claude
                               → Claude returns a SQL query

3. is_safe() + execute()    → validate SQL is SELECT-only
                               → run against DuckDB in read-only mode

4. narrate()                → send (question + result rows) to Claude
                               → Claude writes a 2-3 sentence plain-English answer
```

Two Claude API calls per question: one to write SQL, one to explain the answer.

### Why the schema comes from the dbt catalog

`schema_context.py` reads `dbt/sp500_analytics/target/catalog.json` — the file
`dbt docs generate` produces. This means:

- The LLM sees **exactly the columns that exist** in the current marts, with their
  dbt descriptions as context.
- When you add a new column to a mart and run `dbt docs generate`, the LLM
  automatically knows about it — no code change needed.
- The column descriptions you write in `_marts.yml` become the LLM's "schema
  brain" — good documentation literally makes the AI smarter.

If `catalog.json` isn't present, `schema_context.py` falls back to a static
string so the app still works.

### Safety: two layers

**Layer 1 — Prompt rules:**
```python
SQL_SYSTEM_PROMPT = """...
- Output ONLY the SQL, no prose, no markdown fences.
- SELECT statements only. Never write INSERT/UPDATE/DELETE/DROP/ALTER/CREATE.
- Use ONLY these tables: {tables}.
..."""
```

**Layer 2 — Code validation:**
```python
FORBIDDEN = re.compile(r"\b(insert|update|delete|drop|alter|create|attach|copy|pragma)\b", re.I)

def is_safe(sql: str) -> bool:
    return sql.lower().lstrip().startswith("select") and not FORBIDDEN.search(sql)
```

And the DuckDB connection opens in read-only mode:
```python
con = duckdb.connect(DUCKDB_PATH, read_only=True)
```

Even if the prompt injection somehow made Claude generate a `DROP TABLE`, DuckDB
would reject it because the connection is read-only. Defence in depth.

### Why only `mart_` tables?

`allowed_tables()` returns only the four Gold marts. The Silver views
(`stg_*`, `int_*`) are excluded — they're internal implementation details with
unstable schemas. This keeps the LLM focused and prevents it from generating
joins across staging models that might change.

## Hands-on

Make sure your `.env` has a working `ANTHROPIC_API_KEY`, and that you've run
`dbt docs generate` recently (so `catalog.json` exists).

### 1. Generate the catalog (if not already done)

```powershell
cd dbt\sp500_analytics
$env:DATA_DIR = "C:\dev\GithubFabricProject\data"
$env:DBT_PROFILES_DIR = "C:\dev\GithubFabricProject\dbt\sp500_analytics\ci"
dbt docs generate --target duckdb
```

### 2. Run the app

From the **repo root** (not the dbt folder):

```powershell
cd C:\dev\GithubFabricProject
streamlit run nl_query/app.py
```

Why `streamlit run` and not `python`? Streamlit is a web framework — it starts
its own web server and serves a browser UI. `python nl_query/app.py` would also
work (because of the `if __name__ == "__main__": main()` block) but Streamlit's
hot-reload and layout engine only kick in via `streamlit run`.

A browser tab opens at `http://localhost:8501`.

### 3. Try these questions

Start with these to confirm the pipeline works end-to-end:

```
Which 5 companies had the highest R&D intensity last fiscal year?
```
```
What is the average net margin by GICS sector?
```
```
Show me MSFT's closing price and 50-day moving average for the last 30 trading days.
```

For each question, look at:
- **Generated SQL** — does it reference the right tables and columns?
- **Answer** — is the narrative factually consistent with the result table?
- **Chart** — Streamlit auto-picks the first datetime/text column as x-axis and
  first numeric column as y-axis. It's a best-effort chart, not a designed one.

### 4. Look at the schema the LLM sees

Before typing a question, the app shows an expandable **"Schema the assistant can
see"** section. Open it. This is exactly what Claude receives as context — the
compact column list built from your dbt catalog. Notice that only `mart_*` tables
appear, and that the descriptions you wrote in `_marts.yml` show up here.

### 5. Try to break it

Ask something outside the schema:
```
What is the weather in New York today?
```
```
Show me all users in the database.
```

The SQL will either fail (table doesn't exist) or Claude will correctly say it
can't answer from the available data.

Press `Ctrl+C` in PowerShell to stop the app when done.

## Exercises

1. **Add a column description.** In `_marts.yml`, add a description to
   `mart_prices.ma_50` (e.g. "50-day moving average of adjusted close price").
   Run `dbt docs generate --target duckdb`. Open the app and check the schema
   panel — does the new description appear? Ask "what does ma_50 represent?" —
   does Claude's answer improve?

2. **Trace a generated SQL query.** Ask "Which sector has the highest average
   revenue?" Copy the generated SQL. Open a Python prompt, connect to DuckDB, and
   run the same SQL manually. Do the results match?

3. **Understand the two-call cost.** Every question makes two Claude API calls
   (SQL generation + narration). At Sonnet 4.6 pricing (~$3/M input tokens),
   estimate the cost of 100 questions per day. How would you reduce it?
   (Hint: cache the schema; use a smaller model for narration.)
